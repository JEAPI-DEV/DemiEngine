\chapter{Saves and Settings}

\section{Save File Shape}

Save files are source-like JSON files with a stable format:

\begin{lstlisting}[language=json]
{
  "format_version": 1,
  "slot": "settings",
  "state": {
    "master_volume": 1.0,
    "window_mode": "windowed"
  }
}
\end{lstlisting}

The Lua save service currently supports flat number and string values in the \code{state} object.

\section{Settings Persistence}

The \filepath{minimal\_2d} example uses save slot \code{settings}. On menu startup it loads:

\begin{itemize}
  \item \code{master\_volume};
  \item \code{window\_mode}.
\end{itemize}

When the user changes volume or window mode in the options menu, Lua immediately updates the save file. The runtime applies the loaded values on startup.

\section{Structured Game Saves}

The settings helpers remain available, while \code{Save.write\_state} and
\code{Save.read\_state} provide the explicit game-save boundary. A structured
state contains \code{game}, \code{selected\_entities},
\code{prefab\_instances}, and deliberately nominated \code{lua} tables.
Autosave sequence and reason metadata are available through
\code{Save.metadata}; incompatible versions are rejected with a message from
\code{Save.last\_error}. Registered migration hooks run while reading older
state versions.
